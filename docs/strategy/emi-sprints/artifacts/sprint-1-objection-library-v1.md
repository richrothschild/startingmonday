# Sprint 1 Artifact: Objection Library v1

Date: 2026-05-25
Owner: GTM Lead
Status: Draft for Sprint 1 execution

## Usage Standard

1. Identify segment before response.
2. Acknowledge concern in plain language.
3. Answer with one claim and one proof path.
4. End with one low-friction next step.

## B2C Executive Objections

1. Objection: I do not need this until I am actively searching.
- Response: Most executive search failure starts before search start, when momentum and relationships are unmanaged.
- Proof path: show early-signal and weekly-cadence examples.
- Next step: run EMI assessment and get recommended mode.

2. Objection: I already have a network and recruiter relationships.
- Response: EMI is not a contact list replacement; it is cadence enforcement and signal-aware follow-through.
- Proof path: show action-loop and relationship cadence summary.
- Next step: import top ten relationships and run one weekly plan.

3. Objection: This feels like more work.
- Response: daily loop limits work to three priorities so effort is controlled and measurable.
- Proof path: show one-screen daily plan and completion trend.
- Next step: run seven-day pilot loop.

4. Objection: I am burned out and cannot run a heavy process.
- Response: the system is designed to reduce decision load and remove planning overhead.
- Proof path: show 72-hour recovery protocol and simplified plan mode.
- Next step: enable low-friction recovery mode.

5. Objection: I only want outcomes, not another framework.
- Response: framework exists to improve execution consistency and outcome probability.
- Proof path: show readiness and completion trend metrics.
- Next step: commit to one weekly review and measure change.

## Coach Objections

6. Objection: this will replace coaching.
- Response: the system replaces context rebuild, not coach judgment.
- Proof path: show shared digest and between-session summary.
- Next step: run one client with shared-view workflow.

7. Objection: clients will not use tools between sessions.
- Response: action limits and reminders are designed for low-friction adherence.
- Proof path: show weekly completion pattern from pilot users.
- Next step: enroll one client in 14-day cadence pilot.

8. Objection: this adds admin burden for me.
- Response: counselor and coach views are designed to reduce manual status collection.
- Proof path: show weekly digest replacing manual check-ins.
- Next step: compare one week before and after digest workflow.

## Outplacement Buyer Objections

9. Objection: unclear margin impact for our programs.
- Response: EMI is positioned as improved completion and readiness, not labor-intensive customization.
- Proof path: show pilot ROI calculator by cohort size.
- Next step: run a scoped pilot business case.

10. Objection: rollout may require founder involvement.
- Response: runbook and handoff model is designed for operator-led delivery.
- Proof path: show runbook, roles, and escalation design.
- Next step: validate implementation with operations lead.

11. Objection: legal review risk is too high.
- Response: trust starter pack defines data boundaries, claims policy, and KPI terms.
- Proof path: provide trust center packet during procurement.
- Next step: start legal pre-review with published controls.

## Sales and Partnerships Objections

12. Objection: category language still sounds abstract.
- Response: category is always paired with explicit alternative and concrete daily mechanics.
- Proof path: show category one-liner plus daily loop workflow.
- Next step: run problem-language discovery call script.

13. Objection: no urgency to buy now.
- Response: momentum loss compounds silently and increases transition cost over time.
- Proof path: show pain-to-cost matrix and readiness risk triggers.

- Next step: review one claim with full evidence block.

## Response QA Checklist

1. Segment is identified before response.
2. Response includes one concrete proof path.
3. Next step asks for low-friction commitment.
4. No defensive or inflated language.
